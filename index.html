TopUp & Voucher
Redemption API

Unified REST API for Garena Free Fire top-up and voucher redemption. Supports batch processing, async callbacks, Telegram notifications, and multi-region shops.

🌐 Host 46.250.227.190:8002
⚡ Async FastAPI v4
🔐 Bearer Token
📦 Batch up to 5
01

Introduction

The TopUp & Voucher API provides a single POST /topup endpoint that handles three types of orders determined by the code field:

🎟️

Voucher

Redeem SERIAL:PIN codes on Garena BD via UniPin. Batch up to 5.

🇧🇩

Shell (BD)

Direct top-up on Garena Bangladesh shop. All packages supported.

🇸🇬

Shell (SG)

Direct top-up on Garena Singapore shop. LITE and 30D only.

ℹ️
All orders are processed asynchronously in the background. The API responds immediately with accepted or processing, then delivers results via HTTP callback and Telegram notifications when done.

Authentication

Include your API token using any one of these methods (checked in priority order):

① Header — Bearer
Authorization: Bearer YOUR_TOKEN
② Header — Raw
Authorization: YOUR_TOKEN
③ Query Param
POST /topup?token=YOUR_TOKEN
④ Body Field
"api_key": "YOUR_TOKEN"
⚠️
Each order deducts qty credits from your token balance. Ensure sufficient credits before sending. Insufficient balance returns HTTP 401.

POST /topup

POST /topup Auth required

Unified endpoint for all order types. Order type is determined by the code field value.

Request Body Parameters

FieldTypeRequiredDescription
orderid string required Your unique order identifier
playerid string required Garena player UID (numeric, 8+ digits for vouchers)
code string required "shell" | "sgshell" | SERIAL:PIN[,...]
url string optional Your callback URL for result delivery
api_key string alt-auth Token in body (alternative to Authorization header)
package string shell only Package code e.g. LITE, 30D, 108589
pacakge string alias Typo-friendly alias for package
username string shell only Garena account username
password string shell only Garena account password
autocode string shell only 2FA/secret key for Garena account
qty integer optional Quantity 1–5 (default: 1)
metadata object optional Any extra fields — echoed back in callback response

Callback System

After processing completes, results are delivered via three channels:

1

HTTP POST → your callback URL

A JSON payload is posted to the url field from your request. Simple status/orderid/content format.

2

Telegram → ORDER_CALLBACK_CHANNEL

Detailed result message posted to the configured server Telegram channel. Always fires regardless of url.

3

Telegram → Your Token's Group

If your API token is bound to a Telegram group, the same result is also sent there.

Callback Payload — Success

POST → your url 
{
  "status": "success",
  "orderid": "ORD-001",
  "content": "✅ BDMB-S-S-02971495 4193-5369-5635-5998 (800UC/Monthly)"
}

Callback Payload — Failed

POST → your url
{
  "status": "failed",
  "orderid": "ORD-001",
  "content": "❌ BDMB-S-S-02971495 4193-5369-5635-5998 — Consumed Voucher"
}

Callback Payload — Partial (Batch)

POST → your url
{
  "status": "partial",
  "orderid": "ORD-001",
  "content": {
    "batch": [
      {
        "uc": "BDMB-S-S-02971495 4193-5369-5635-5998",
        "ok": true,
        "detail": "✅ Success"
      },
      {
        "uc": "BDMB-J-S-11111111 1111-2222-3333-4444",
        "ok": false,
        "detail": "❌ Consumed Voucher"
      }
    ]
  }
}

Callback with Metadata

If you include a metadata object in your request, it is echoed back in the callback:

REQUEST with metadata
{
  "orderid": "ORD-005",
  "playerid": "2716319203",
  "code": "BDMB-S-S-02971495:4193536956355998",
  "url": "https://yourserver.com/callback",
  "metadata": {
    "customer_id": "user_123",
    "ref": "invoice_456"
  }
}
CALLBACK includes metadata
{
  "status": "success",
  "orderid": "ORD-005",
  "content": "✅ BDMB-S-S-02971495 ...",
  "metadata": {
    "customer_id": "user_123",
    "ref": "invoice_456"
  }
}
ℹ️
Your callback server must return HTTP 200. Callback delivery is attempted once — no automatic retry on failure. Ensure your endpoint is publicly reachable.

import asyncio
import aiohttp

API_URL = "http://46.250.227.190:8002"
TOKEN   = "YOUR_TOKEN"

async def topup(payload: dict) -> dict:
    headers = {
        "Authorization": f"Bearer {TOKEN}",
        "Content-Type":  "application/json",
    }
    async with aiohttp.ClientSession() as session:
        async with session.post(
            f"{API_URL}/topup",
            json=payload,
            headers=headers,
            timeout=aiohttp.ClientTimeout(total=30)
        ) as resp:
            resp.raise_for_status()
            return await resp.json()

async def main():
    result = await topup({
        "orderid":  "ORD-001",
        "playerid": "2716319203",
        "code":     "BDMB-S-S-02971495:4193536956355998",
        "url":      "https://yourserver.com/callback",
    })
    print(result)

asyncio.run(main())

Go Integration

Go (net/http) 
package main

import (
    "bytes"
    "encoding/json"
    "fmt"
    "net/http"
    "time"
)

const (
    APIURL = "http://46.250.227.190:8002"
    TOKEN  = "YOUR_TOKEN"
)

type TopUpRequest struct {
    OrderID  string `json:"orderid"`
    PlayerID string `json:"playerid"`
    Code     string `json:"code"`
    URL      string `json:"url,omitempty"`
    Package  string `json:"package,omitempty"`
    Username string `json:"username,omitempty"`
    Password string `json:"password,omitempty"`
    Autocode string `json:"autocode,omitempty"`
}

func SendTopUp(req TopUpRequest) (map[string]interface{}, error) {
    body, _ := json.Marshal(req)
    httpReq, _ := http.NewRequest("POST", APIURL+"/topup", bytes.NewBuffer(body))
    httpReq.Header.Set("Authorization", "Bearer "+TOKEN)
    httpReq.Header.Set("Content-Type", "application/json")

    client := &http.Client{Timeout: 30 * time.Second}
    resp, err := client.Do(httpReq)
    if err != nil { return nil, err }
    defer resp.Body.Close()

    var result map[string]interface{}
    json.NewDecoder(resp.Body).Decode(&result)
    return result, nil
}

func main() {
    res, err := SendTopUp(TopUpRequest{
        OrderID:  "ORD-001",
        PlayerID: "2716319203",
        Code:     "BDMB-S-S-02971495:4193536956355998",
        URL:      "https://yourserver.com/callback",
    })
    if err != nil { panic(err) }
    fmt.Println(res["status"], res["orderid"])
}

Best Practices

✅
Always provide a callback URL. The API is async — you will not receive the result in the HTTP response. Without a callback URL, you have no way to know if an order succeeded.
💡
Use metadata for order tracking. Pass "metadata": {"ref": "your_internal_id"} — it's echoed back in callbacks so you can match results to your internal orders without parsing order IDs.
⚠️
Idempotency. Use unique orderid values per order. The API does not deduplicate by orderid — duplicate submissions will each consume credits and process independently.
🔒
Token security. Never expose your API token in client-side code. Always make API calls from your backend server. Rotate your token if you suspect it has been leaked.
⚡
Batch vouchers for speed. Sending 5 vouchers in one order (comma-separated) is faster than 5 separate orders — they share one login session and process in parallel.
🔁
Callback retries. The API does not retry failed callback deliveries. Implement your own retry logic or polling if your callback server has downtime.
